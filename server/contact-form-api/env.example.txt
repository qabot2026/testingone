# WHAT THIS DOES
# --------------
# When someone fills the chat contact form, your server sends ONE email:
#
#   FROM = your company mailbox (the "sender" people see)
#   TO   = your staff / client inboxes (who receives the lead)
#   CC   = extra people who also get a copy (optional)
#
# The visitor's NAME, EMAIL, and PHONE are inside the message text (not separate).
#
# WHERE YOU TYPE THIS
# -------------------
# - Live site: Railway → your API service → Variables → add each line below.
# - Local test: create file  server/contact-form-api/.env   (same folder as index.mjs)
#
# =============================================================================

# --- Who receives the lead (main addresses) ---
# One or more emails, separated by commas.
CONTACT_LEAD_NOTIFY_TO=office@yourcompany.com

# --- Extra copy (optional) ---
# One or more emails, separated by commas. Leave empty if you do not need CC.
CONTACT_LEAD_NOTIFY_CC=

# --- Sender ("From") — usually your company address ---
MAIL_FROM=noreply@yourcompany.com

# --- SMTP login (same provider as your company email, e.g. Gmail / Microsoft) ---
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=noreply@yourcompany.com
SMTP_PASS=your-app-password-here

# Gmail: use an "App password", not your normal Gmail password.
# Port 465 SSL: set SMTP_PORT=465 and SMTP_SECURE=1
SMTP_SECURE=

# Optional: change the start of the email subject line
CONTACT_LEAD_NOTIFY_SUBJECT=New website lead
#
# Staff inbox — new-lead notification HTML (`appointment_mail_to_user.html` was `staff-new-lead-simple.html`):
# CONTACT_LEAD_STAFF_MAIL_TEMPLATE=appointment_mail_to_user.html
# CONTACT_LEAD_STAFF_MAIL_INTRO=A new lead has been captured from the chatbot.
# CONTACT_LEAD_STAFF_LEAD_SOURCE_LABEL=   # replaces auto “Source:” text (plain)
# CONTACT_LEAD_STAFF_LEAD_SOURCE_FALLBACK=Website Chatbot   # used when technical source slug is generic
# CONTACT_MAIL_COMPANY_NAME=Your Clinic Name
# CONTACT_MAIL_COMPANY_WEBSITE=www.yoursite.com   # footer line (protocol optional — stripped on send)

# After deploy, open in browser (use your Railway URL):
#   GET https://YOUR-API.railway.app/contact-form-email-health
# If "missing_env" is not empty, those variables are not set on the server.
#
# Optional: set CONTACT_LEAD_EMAIL_DEBUG=1 to log every send/skip in Railway Logs.
# Delay before SMTP send, after HTTP 200 (+ deferred Firestore in tail). Omit for default 60000 (1 min); use 0 to send ASAP after response flush.
# CONTACT_LEAD_EMAIL_DELAY_MS=60000
# Optional (debug): CONTACT_LEAD_ATTACH_OUTCOME_IN_JSON=1 adds a non-blocking `lead_email.scheduled` hint to submission JSON — does NOT wait on SMTP.
#
# Boot SMTP check + send limits (cold Railway → slow TLS often needs higher budgets).
# CONTACT_LEAD_SMTP_CONNECT_TIMEOUT_MS defaults 45000 (TCP+TLS to Gmail).
# CONTACT_LEAD_SMTP_VERIFY_TIMEOUT_MS defaults 45000 — try 60000 if verify times out.
# CONTACT_LEAD_SEND_TIMEOUT_MS defaults 180000 (whole sendMail); max 300000. Lower only if you want faster fail.
# Gmail "Connection timeout" on Railway: SMTP_DNS_IPV4_FIRST=1 (default) prefers IPv4 over IPv6.
# If 465 still hangs, use SMTP_PORT=587 and leave SMTP_SECURE empty (STARTTLS).

# Endpoints added by mountSmsOtpRoutes(app) in index.mjs:
#   POST /api/sms-otp/send    { "mobile": "9999999999" }
#   POST /api/sms-otp/verify  { "mobile": "9999999999", "code": "123456" }
#   GET  /api/sms-otp/health
#
# Required: get your dev key from https://www.fast2sms.com/dashboard/dev-api
# FAST2SMS_API_KEY=your_fast2sms_dev_api_key_here
#
# Optional — route selection. Default "otp" needs no DLT approval and sends
# Fast2SMS's generic "Your OTP verification code is XXXXXX" message.
#   otp : generic message, no template needed (recommended for getting started)
#   q   : quick transactional, custom message via FAST2SMS_MESSAGE_TEMPLATE
#   v3  : sender-id route, needs FAST2SMS_SENDER_ID
#   dlt : DLT route, needs FAST2SMS_SENDER_ID + FAST2SMS_DLT_MESSAGE_ID
# FAST2SMS_ROUTE=otp
# FAST2SMS_SENDER_ID=
# FAST2SMS_DLT_MESSAGE_ID=
# FAST2SMS_MESSAGE_TEMPLATE=Your OTP is {{otp}}
#
# Optional — OTP behaviour tuning:
# SMS_OTP_LENGTH=6                # 4..8, default 6
# SMS_OTP_TTL_SECONDS=300         # 30..900, default 300 (5 minutes)
# SMS_OTP_MAX_ATTEMPTS=5          # 1..20, default 5 verify failures before lockout
# SMS_OTP_RESEND_COOLDOWN_S=30    # 0..600, default 30 seconds between sends per mobile
#
# After deploy, verify configuration in the browser:
#   GET https://YOUR-API.railway.app/api/sms-otp/health
